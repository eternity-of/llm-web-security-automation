# Business Logic Diagnosis Playbook

이 문서는 기술 취약점 이름으로 바로 분류되지 않는 비즈니스 로직 취약점을 찾기 위한 기준이다.

## 1. 상태 전이 검증

확인 질문:

- 선행 단계를 거치지 않고 후속 단계를 호출할 수 있는가?
- 선행 단계 실패 후에도 후속 단계가 성공하는가?
- 이미 완료된 요청을 재전송하면 중복 처리되는가?
- 취소된 객체를 다시 취소/수정/조회할 수 있는가?
- 승인 전 상태의 객체를 승인 후 기능으로 조작할 수 있는가?
- 임시 토큰, 인증번호, 예약, 쿠폰, 초대 링크가 재사용 가능한가?

고위험 흐름:

- 비밀번호 재설정
- 이메일/휴대폰 변경
- 2FA/OTP 등록
- 주문/결제/환불
- 예약/취소
- 승인/반려
- 회원가입/초대/계정 복구

## 2. 권한 경계 검증

확인 질문:

- 일반 사용자 세션으로 관리자 요청을 재전송할 수 있는가?
- 화면에 버튼이 없어도 API 직접 호출이 가능한가?
- 다른 사용자의 객체를 조회/수정/삭제/다운로드할 수 있는가?
- tenant, company, org, department 경계를 넘을 수 있는가?
- 권한 없는 요청에 200 OK와 정상 JSON 구조가 반환되는가?

## 3. 객체 소유권 검증

다음 객체는 소유권 검증 대상이다.

- 게시글 / 댓글
- 파일 / 이미지 / 첨부파일
- 예약 / 주문 / 결제 / 환불
- 문의 / 문서 / 알림
- 사용자 프로필 / 설정
- 관리자 승인 대상 객체

확인 질문:

- 객체 ID가 순차값인가?
- 목록에 없는 객체를 직접 접근할 수 있는가?
- 조회는 막히지만 다운로드/삭제는 가능한가?
- 수정/삭제 요청에서 소유자를 서버가 재확인하는가?
- 객체의 공개/비공개 상태가 다운로드에도 적용되는가?

## 4. 클라이언트 신뢰 검증

위험한 입력값:

- userId, memberNo, writer, owner
- role, isAdmin, permission
- price, amount, discount, point
- status, approval, step
- room, time, reservationId
- filePath, fileName, savePath
- redirectUrl, callback, endpoint

확인 질문:

- 사용자가 값을 바꾸면 서버 결과도 바뀌는가?
- 서버가 세션/DB 기준으로 값을 재계산하는가?
- hidden field, disabled field, JSON field가 신뢰되는가?
- Content-Type 변경 시 검증이 달라지는가?

## 5. 반복/재사용 검증

확인 질문:

- 동일 요청 재전송 시 중복 처리되는가?
- 같은 토큰을 여러 번 사용할 수 있는가?
- 같은 인증번호로 여러 계정/기능을 처리할 수 있는가?
- 같은 파일명/예약시간/쿠폰이 중복 사용되는가?

## 6. 오류 처리와 실제 상태 불일치

확인 질문:

- 실패 메시지가 반환됐는데 실제 상태가 변경되는가?
- 성공 메시지가 반환됐는데 실제 상태가 변경되지 않는가?
- 에러 응답에 다음 단계 진행에 필요한 상태가 세션에 남는가?
- 비정상 입력 후 세션/토큰/객체 상태가 오염되는가?

## 7. 보고서 작성 기준

비즈니스 로직 취약점은 다음을 반드시 포함한다.

- 정상 흐름
- 비정상 흐름
- 서버가 검증해야 하는 조건
- 실제로 검증하지 않은 조건
- 악용에 필요한 권한/계정 조건
- 실제 상태 변경 여부
- 동일 패턴 확장 가능성
