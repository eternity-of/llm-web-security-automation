# Payload Library

이 문서는 Burp Suite MCP 기반 웹 취약점 진단 중 사용할 저위험 테스트 문자열과 payload 변형 규칙을 정의한다.

목적은 취약점 악용이 아니라 다음 항목을 확인하는 것이다.

- 입력값 반영 여부
- 출력 인코딩 여부
- 필터링 여부
- 타입 검증 여부
- 응답 차이 여부
- 에러 메시지 노출 여부

모든 테스트는 `safety_policy.md` 기준을 따른다.

---

## 1. 공통 원칙

- 허가된 Scope 내에서만 사용한다.
- 상태 변경 요청에는 사용자 승인 없이 사용하지 않는다.
- 대량 반복 테스트에 사용하지 않는다.
- 데이터 추출, 권한 상승, 인증 우회 실사용, 서비스 부하 유발 목적에 사용하지 않는다.
- 원본 요청과 원본 응답을 먼저 확보한 뒤, 변조 요청과 비교한다.
- 민감정보가 노출되면 즉시 중단한다.
- 자동화는 기본적으로 단일 요청 또는 최소 요청 수로 수행한다.

---

## 2. 공통 Marker

다음 marker는 입력값 반영 여부 확인에 사용한다.

```text
security_probe_{{RAND}}
security_PROBE_{{RAND}}
security-probe-{{RAND}}
security_probe_{{RAND}}_end
```

판단 기준:

- 응답 본문에 marker가 그대로 반영되는가
- HTML Entity Encoding이 적용되는가
- 일부 문자만 제거되는가
- 저장 후 다른 화면에서 재출력되는가
- 반영 위치가 HTML 본문, Attribute, JavaScript 문자열, URL 중 어디인가

---

## 3. XSS 반영 위치 확인용

목적:

- 스크립트 실행이 아니라 반영 위치와 인코딩 여부를 확인한다.
- HTML 본문, Attribute, JavaScript 문자열, URL 영역 중 어디에 반영되는지 분류한다.

```text
xss_probe_{{RAND}}
xss_probe_{{RAND}}<>
xss_probe_{{RAND}}'"
xss_probe_{{RAND}}`"
"><xss_probe_{{RAND}}
'><xss_probe_{{RAND}}
&lt;xss_probe_{{RAND}}&gt;
%3Cxss_probe_{{RAND}}%3E
```

판단 기준:

- `<`, `>`, `"`, `'`, `` ` `` 문자가 인코딩되는가
- 입력값이 HTML 본문에 반영되는가
- 입력값이 태그 속성 내부에 반영되는가
- 입력값이 JavaScript 문자열 내부에 반영되는가
- 저장 후 목록/상세/관리자 화면에 재출력되는가

주의:

- 실제 스크립트 실행 payload는 사용자 승인 없이 사용하지 않는다.
- 저장형 테스트는 테스트 계정과 테스트 데이터에서만 수행한다.
- 관리자 또는 실제 사용자 화면을 대상으로 실행하지 않는다.

---

## 4. SQL Injection 저위험 오류 확인용

목적:

- DB 에러 노출 또는 응답 차이를 확인한다.
- 데이터 추출은 사용자 허가 하에 진행한다.

```text
'
"
`
)
')
")
test'
test"
{{RAND}}'
{{RAND}}"
' AND '1'='1
' AND '1'='2
" AND "1"="1
" AND "1"="2
```

판단 기준:

- 원본 요청 대비 Status Code가 달라지는가
- Response Length가 크게 달라지는가
- SQL 문법 오류가 노출되는가
- DBMS 이름, 테이블명, 컬럼명, SQL 일부가 노출되는가
- 단순 500 에러인지 DB 에러인지 구분 가능한가

금지:

- time-based 반복 테스트
- DB 쓰기/수정/삭제 목적 구문
- WAF 우회 목적의 대량 변형

---

## 5. NoSQL Injection 구조 확인용

목적:

- JSON 필드 타입 검증 여부를 확인한다.
- 조건 우회나 대량 조회는 수행하지 않는다.

확인 관점:

```text
문자열 필드 → 숫자
문자열 필드 → 배열
문자열 필드 → 객체
숫자 필드 → 문자열
필수 필드 제거
알 수 없는 필드 추가
```

판단 기준:

- JSON 파싱 에러가 발생하는가
- 서버가 타입 오류를 명확히 차단하는가
- 응답 데이터 범위가 비정상적으로 넓어지는가
- 인증/검색 조건이 우회되는 징후가 있는가

중단 조건:

- 민감 데이터가 다량 반환됨
- 원래보다 현저히 넓은 데이터 범위가 반환됨
- 인증 우회 가능성이 확인됨

---

## 6. Open Redirect 확인용

목적:

- 리다이렉트 파라미터가 사용자 입력을 신뢰하는지 확인한다.

```text
/test-redirect-probe
https://example.com/
%2Ftest-redirect-probe
```

판단 기준:

- Location 헤더가 입력값으로 설정되는가
- 외부 도메인으로 이동 가능한가
- 상대경로와 절대 URL을 구분하는가
- 로그인 후 returnUrl에 적용되는가

주의:

- 실제 피싱 페이지 또는 악성 도메인은 사용하지 않는다.
- 사용자에게 링크를 전송하지 않는다.

---

## 7. 파일 다운로드 확인용

목적:

- 인증/인가 검증 여부를 확인한다.
- 경로 탐색이나 대량 다운로드는 수행하지 않는다.

확인 관점:

```text
Cookie 제거
Authorization 제거
동일 fileId 재요청
테스트 계정 간 허가된 fileId 단일 비교
Content-Disposition 확인
Content-Length 확인
```

판단 기준:

- 인증 제거 후에도 파일이 반환되는가
- 권한 없는 계정으로 동일 파일이 반환되는가
- 파일명, 파일 크기, Content-Disposition이 동일한가
- 게시글/객체 조회 권한과 다운로드 권한이 분리되어 있는가

금지:

- 대량 fileId 순회
- 서버 내부 파일 탐색
- 운영 첨부파일 대량 수집

---

## 8. CSRF 확인용

목적:

- 상태 변경 요청에 CSRF 방어가 존재하는지 확인한다.
- 실제 상태 변경은 사용자 승인 후 테스트 데이터에서만 확인한다.

확인 관점:

```text
CSRF Token 존재 여부
Token 제거 시 응답 차이
Origin / Referer 검증 여부
SameSite Cookie 속성
GET 기반 상태 변경 여부
```

판단 기준:

- Token이 없는가
- Token 제거 후에도 요청이 성공하는가
- Origin / Referer가 없거나 외부여도 요청이 성공하는가
- GET 요청만으로 상태 변경이 가능한가

주의:

- 결제, 탈퇴, 권한 변경, 비밀번호 변경 등 민감 기능에는 자동 테스트하지 않는다.
- 테스트 데이터에 대해서만 수행한다.

---

## 9. Payload 변형 규칙

Claude는 테스트 문자열이 필요한 경우 아래 순서로만 변형한다.

1. 원본 marker 사용
2. 대소문자 변형
3. 구분자 추가
4. 따옴표 추가
5. URL 인코딩 1회
6. HTML Entity 인코딩 1회

자동화에서 금지되는 변형:

- 이중/다중 인코딩 반복
- WAF 우회 목적의 대량 변형
- time-based 변형
- 데이터 추출 목적 변형
- 공격 성공 가능성을 높이기 위한 반복 payload 생성
- 서비스 장애를 유발할 수 있는 변형
