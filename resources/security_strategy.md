# Advanced Web security Strategy

이 문서는 단순 체크리스트 탐지를 넘어 서비스 흐름 기반 취약점 가설을 만들기 위한 전략이다.

## 1. 기본 원칙

- `web_checklist.md`는 최소 기준이며 최종 진단 범위가 아니다.
- 요청을 개별 URL이 아니라 기능 흐름의 일부로 해석한다.
- 단일 취약점명보다 근본 원인과 기능 간 영향을 우선한다.
- 확인한 사실, 강한 추론, 추가 확인 필요를 항상 분리한다.

## 2. 모든 기능에 던질 질문

1. 이 기능이 다루는 자산은 무엇인가?
2. 이 기능을 사용할 수 있는 주체는 누구인가?
3. 이 요청은 서버 상태를 변경하는가?
4. 서버가 클라이언트 입력값 중 무엇을 신뢰하는가?
5. 객체의 소유자는 누구인가?
6. 현재 사용자와 객체 소유자의 관계를 서버가 검증하는가?
7. 이전 단계의 검증 결과가 다음 단계에서 재검증되는가?
8. 실패해야 할 요청이 성공하는 경우가 있는가?
9. 실패 응답인데 실제 상태가 변경되는가?
10. 단일 기능이 다른 기능과 결합될 때 영향이 커지는가?

## 3. 서비스 모델링

먼저 다음 항목을 만든다.

```text
Actor:
Asset:
Action:
State:
Trust Boundary:
Permission Boundary:
```

예시 Actor:

- 비로그인 사용자
- 일반 사용자
- 관리자
- 외부 연동 시스템

예시 Asset:

- 계정
- 세션
- 게시글
- 댓글
- 첨부파일
- 예약
- 주문
- 결제
- 권한
- 설정

예시 Trust Boundary:

- 사용자 입력 파라미터
- Cookie / Authorization Header
- 서버 세션
- 데이터베이스 값
- 파일 시스템
- 외부 URL
- 관리자 기능

## 4. 상태 전이 분석

다음 구조는 우선순위가 높다.

- 1단계 검증 후 2단계 상태 변경
- 인증번호/토큰 발급 후 검증
- 예약 생성 후 취소
- 결제 생성 후 취소/환불
- 게시글 작성 후 수정/삭제
- 파일 업로드 후 다운로드
- 관리자 승인/반려

분석 질문:

- 후속 단계를 직접 호출할 수 있는가?
- 실패한 선행 단계 이후 후속 요청이 가능한가?
- 완료된 요청을 재전송하면 중복 처리되는가?
- 취소/삭제된 객체를 다시 조작할 수 있는가?
- 임시 토큰이 일회성인가?
- 세션 상태가 실패/성공 조건과 일치하는가?

## 5. 권한 경계 분석

권한은 화면 표시가 아니라 서버에서 검증되어야 한다.

확인 질문:

- 일반 사용자 세션으로 관리자 요청을 재전송하면 어떻게 되는가?
- 버튼이 없어도 API 직접 호출이 가능한가?
- 같은 객체 ID를 다른 사용자 세션에서 사용할 수 있는가?
- 조직/부서/회사/tenant 경계를 넘을 수 있는가?
- 조회는 막히지만 다운로드/삭제는 가능한 기능이 있는가?

## 6. 신뢰 경계 분석

다음 클라이언트 입력값은 서버가 그대로 신뢰하면 위험하다.

- userId / memberNo / accountId
- role / isAdmin / permission
- owner / writer / creator
- price / amount / point / discount
- status / approval / step
- filePath / fileName / fileId
- redirectUrl / callback / url
- room / time / reservationId
- companyId / orgId / tenantId

확인 질문:

- 서버가 세션 기준으로 값을 재계산하는가?
- hidden input 변경으로 결과가 달라지는가?
- role/status/owner를 클라이언트가 지정하는가?
- 요청 순서를 바꾸면 검증이 우회되는가?

## 7. 기능 연계 분석

단일 취약점이 낮은 위험도여도 다른 기능과 결합되면 위험도가 높아질 수 있다.

연계 예시:

- Stored XSS + CSRF
- 정보 노출 + IDOR
- 인증 없는 파일 다운로드 + 순차 ID
- Open Redirect + 로그인/인증 메일
- 비밀번호 재설정 흐름 미흡 + 관리자 계정
- 파일 업로드 경로 노출 + 파일 직접 접근

## 8. 우선순위 산정

우선순위를 높이는 조건:

- 비인증 접근 가능
- 타 사용자 데이터 접근 가능
- 상태 변경 가능
- 관리자 기능 영향
- 민감정보 포함
- 반복 악용 가능
- 기능 간 연계 가능
- 동일 패턴이 여러 기능에 존재

## 9. 판단 규칙

- 직접 확인하지 않은 내용은 Confirmed로 표시하지 않는다.
- HTML 태그 렌더링만 확인한 경우 Stored HTML Injection 또는 XSS 가능성으로 구분한다.
- CSRF 토큰 부재만 확인하고 실제 cross-site 성공을 확인하지 않았다면 Partially Confirmed 또는 Inferred로 표시한다.
- 성공 메시지만 있고 실제 상태 변경을 확인하지 못했다면 Partially Confirmed로 표시한다.
