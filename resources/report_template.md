# Web Vulnerability Report Template

이 템플릿은 Burp Suite MCP 기반 웹 취약점 진단 결과를 보고서 형태로 정리하기 위한 기준이다.

보고서는 근거 기반으로 작성한다. 확인한 사실, 강한 추론, 추가 확인 필요를 반드시 분리한다. 확인되지 않은 내용은 단정하지 않는다.

보고서에는 전체 Cookie, Authorization Token, 세션 토큰, 비밀번호, API Key, 개인정보, 파일 전체 내용, 대량 응답 데이터를 포함하지 않는다.

## 위험도 기준

- Critical: 비인증 또는 낮은 권한으로 계정 탈취, 관리자 기능 조작, 민감정보 대량 접근, 중요 상태 변경이 가능한 경우
- High: 타 사용자 데이터 접근/수정/삭제, 권한 없는 파일 다운로드, 중요 기능 인가 미흡, Stored XSS 연계 가능성이 큰 경우
- Medium: 제한적 정보 노출, CSRF 방어 미흡, 일부 권한 검증 미흡, Reflected XSS 가능성, Open Redirect 등
- Low: 보안 헤더 미흡, 낮은 민감도 정보 노출, 설정 개선 사항
- Info: 보안상 참고 사항

## 확인 상태 기준

- Confirmed: 실제 요청/응답 또는 상태 변경으로 확인됨
- Partially Confirmed: 일부 조건은 확인되었으나 최종 악용 가능성은 추가 검증 필요
- Inferred: 구조상 가능성이 높으나 실제 실행하지 않음
- Need Manual Verification: 추가 계정, 권한, 화면 확인, 수동 검증 필요

## 10. 조치 방안

### 공통 조치

- 모든 민감 기능에 서버 측 인증/인가 검증을 적용한다.
- 객체 조회/수정/삭제/다운로드 시 세션 사용자와 객체 소유자/권한을 검증한다.
- 상태 변경 요청은 정상 선행 조건과 현재 상태를 서버에서 검증한다.
- 클라이언트가 전달한 userId, role, status, price, owner, filePath 등을 신뢰하지 않는다.
- 사용자 입력값은 저장/출력 위치에 맞게 검증하고 인코딩한다.
- 에러 메시지에 내부 경로, SQL 구문, 스택트레이스를 노출하지 않는다.

### 취약점별 조치

- 인가 미흡 / IDOR:
  - 객체별 소유자 및 권한 검증을 서버 측에 구현한다.
  - 권한 없는 요청은 403 또는 명확한 오류를 반환한다.

- 상태 전이 / 비즈니스 로직:
  - 후속 단계에서 선행 검증 결과를 재검증한다.
  - 임시 토큰은 계정/기능/만료시간에 바인딩하고 일회성으로 처리한다.

- XSS / HTML Injection:
  - 출력 위치별 Context-aware Encoding을 적용한다.
  - HTML 허용이 필요한 경우 허용 목록 기반 Sanitizing을 적용한다.

- SQL Injection:
  - Prepared Statement 또는 Parameterized Query를 사용한다.
  - 정렬/필터 파라미터는 허용 목록으로 검증한다.

- 파일 다운로드:
  - 파일 식별자와 사용자 권한을 서버 측에서 검증한다.
  - 파일 경로를 사용자 입력으로 직접 받지 않는다.

- 파일 업로드:
  - 확장자, MIME Type, 파일 시그니처를 서버 측에서 검증한다.
  - 업로드 파일은 실행 권한이 없는 경로에 저장한다.

- CSRF:
  - 상태 변경 요청은 POST 등 적절한 메서드를 사용한다.
  - CSRF 토큰과 SameSite 쿠키 속성, Origin/Referer 검증을 적용한다.
