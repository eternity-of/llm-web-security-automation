# Bypass Payload Library

이 문서는 필터링, 인코딩, 정규식 검증, WAF, 서버 측 입력값 검증 로직의 동작 차이를 확인하기 위한 우회 테스트 기준이다.

이 문서는 기본 자동 테스트에 사용하지 않는다. Claude는 사용자가 명시적으로 승인한 경우에만 이 문서를 참고할 수 있다.

---

## 1. 기본 원칙

- 자동 반복 테스트에 사용하지 않는다.
- 사용자 승인 후 단일 요청 단위로만 사용한다.
- 상태 변경 요청에는 기본적으로 사용하지 않는다.
- 데이터 추출, 인증 우회 실사용, 권한 상승, 서비스 부하 유발 목적에 사용하지 않는다.
- WAF 우회 목적의 대량 payload 변형은 수행하지 않는다.
- 민감정보가 노출되면 즉시 중단한다.
- `safety_policy.md`의 Level 3 금지 요청에 해당하면 수행하지 않는다.

---

## 2. 사용 가능 조건

Claude는 다음 조건을 모두 만족할 때만 이 문서를 참고한다.

- 사용자가 우회 테스트를 명시적으로 승인함
- 테스트 대상 요청이 Scope 내에 있음
- 요청 횟수가 제한되어 있음
- 상태 변경 가능성이 낮거나 테스트 데이터만 사용함
- `safety_policy.md`에서 금지 행위에 해당하지 않음
- 테스트 목적이 필터링/인코딩/검증 로직 차이 확인임

---

## 3. 승인 요청 형식

우회 테스트가 필요하다고 판단되면 다음 형식으로 먼저 사용자에게 승인 요청한다.

```text
[우회 테스트 승인 요청]

진단 대상:
관련 기능 흐름:
요청 Method / Path:
취약점 후보:
우회 테스트가 필요한 이유:
대상 파라미터:
사용할 우회 관점:
예상 정상 응답:
취약할 경우 예상 응답:
위험도:
상태 변경 가능성:
민감정보 노출 가능성:
요청 예상 횟수:
중단 조건:
승인 필요 여부:
```

---

## 4. XSS 우회 관점

목적:

- 출력 인코딩이 컨텍스트별로 적용되는지 확인한다.
- 필터가 단순 문자열 제거에 의존하는지 확인한다.
- 실제 피해를 유발하는 스크립트 실행은 수행하지 않는다.

승인 후 검토 가능한 관점:

- 대소문자 변형
- 공백 문자 처리 차이
- HTML Entity 인코딩 처리
- URL 인코딩 처리
- Attribute 컨텍스트 이탈 가능성
- JavaScript 문자열 컨텍스트 이탈 가능성
- 허용 태그 기반 Sanitizer 동작 확인
- 이벤트 핸들러 필터링 여부
- 프로토콜 필터링 여부

금지:

- 쿠키 탈취 코드
- 외부 서버로 데이터 전송
- 피싱 UI 삽입
- 키로깅
- 관리자 또는 실제 사용자 대상 실행
- 저장형 XSS 대량 테스트

결과 정리 기준:

- 어떤 컨텍스트에 반영되었는가
- 어떤 문자가 제거/인코딩되었는가
- 필터링이 서버 측인지 클라이언트 측인지 추정 가능한가
- 실행 가능성은 확인했는가, 아니면 추론인가
- 확인 상태가 Confirmed, Partially Confirmed, Inferred, Need Manual Verification 중 무엇인가

---

## 5. SQL Injection 우회 관점

목적:

- 단순 필터링 또는 escaping 미흡 여부를 확인한다.
- 데이터 추출은 시도하지 않는다.

승인 후 검토 가능한 관점:

- 공백 문자 처리 차이
- 주석 처리 방식에 따른 응답 차이
- 대소문자 변형에 따른 응답 차이
- URL 인코딩 처리 차이
- 숫자형/문자형 타입 차이
- 괄호 처리 차이
- 정렬 파라미터 허용 목록 검증 여부
- 검색 파라미터 escaping 여부
- DB 에러 메시지 노출 여부

금지:

- 데이터 추출
- UNION 기반 컬럼/테이블 식별
- 계정정보 조회
- 파일 읽기
- DB 쓰기
- 데이터 수정/삭제
- 반복적인 time-based 요청
- WAF 우회 목적 대량 payload
- 서비스 부하 유발 쿼리

결과 정리 기준:

- 원본 요청 대비 Status Code 차이
- Response Length 차이
- DB 에러 노출 여부
- 동일 파라미터의 반복 재현성
- 단순 에러인지 SQL 관련 에러인지 구분
- 실제 취약점 확인인지, 우회 가능성 추론인지 분리

---

## 6. NoSQL Injection 우회 관점

목적:

- JSON 구조, 타입 검증, 조건 처리 방식의 차이를 확인한다.
- 인증 우회 또는 대량 데이터 조회는 수행하지 않는다.

승인 후 검토 가능한 관점:

- 문자열 필드의 타입 변경
- 배열/객체 입력 시 처리 방식
- 필수 필드 제거 시 처리 방식
- 알 수 없는 필드 추가 시 처리 방식
- 조건 객체 형태 입력에 대한 차단 여부
- JSON 파싱 에러와 DB 조건 처리 에러 구분

금지:

- 인증 우회 실사용
- 대량 데이터 조회
- 운영 사용자 데이터 수집
- 민감정보 접근
- 반복적인 우회 payload 테스트

결과 정리 기준:

- 입력 타입 변경에 따른 응답 차이
- 데이터 범위가 넓어졌는지 여부
- 서버 측 타입 검증 여부
- 에러 메시지 노출 여부

---

## 7. Open Redirect 우회 관점

목적:

- URL 검증이 허용 목록 기반인지 확인한다.

승인 후 검토 가능한 관점:

- 상대 경로 처리
- 절대 URL 처리
- URL 인코딩 처리
- path confusion 처리
- 허용 도메인 suffix/prefix 검증 여부
- 로그인 후 returnUrl 검증 여부

금지:

- 실제 피싱 페이지 연결
- 사용자에게 링크 전송
- 악성 도메인 사용
- 실제 계정 탈취 시나리오 실행

결과 정리 기준:

- Location 헤더가 어떻게 설정되는가
- 외부 도메인 이동 가능성이 확인되었는가
- 검증이 prefix/suffix 방식으로 보이는가
- 로그인 또는 인증 흐름과 연결되는가

---

## 8. 파일 다운로드 우회 관점

목적:

- 파일 식별자와 권한 검증 관계를 확인한다.
- 경로 탐색이나 대량 수집은 수행하지 않는다.

승인 후 검토 가능한 관점:

- fileId 단일 변경
- 테스트 계정 간 fileId 단일 비교
- 파일명 인코딩 처리
- 경로 구분자 처리
- 확장자 검증 여부
- 게시글 권한과 파일 권한의 분리 여부

금지:

- 대량 fileId 순회
- 서버 내부 파일 탐색
- 경로 순회 자동화
- 민감 파일 전체 다운로드
- 운영 첨부파일 대량 수집

결과 정리 기준:

- 인증 제거 또는 권한 없는 계정으로 파일이 반환되는가
- Content-Disposition, Content-Length가 동일한가
- 게시글 접근은 차단되는데 파일 다운로드는 가능한가
- 최소 증거만 확보했는가

---

## 9. SSRF 우회 관점

목적:

- 서버가 사용자 입력 URL을 직접 요청하는지 확인한다.
- 내부망 접근은 수행하지 않는다.

승인 후 검토 가능한 관점:

- 사용자가 통제하는 테스트 도메인으로 단일 요청
- 허가된 OAST 또는 Collaborator payload 단일 사용
- 리다이렉트 처리 여부
- 허용 도메인 검증 여부
- URL 파싱 방식 차이 확인

금지:

- localhost 접근
- 내부망 접근
- 클라우드 메타데이터 접근
- 포트 스캔
- 내부 서비스 탐색
- 대량 URL 요청
- 민감 내부 시스템 접근

결과 정리 기준:

- 서버 측 요청 발생 근거가 있는가
- 응답 차이가 있는가
- 외부 콜백이 단일 요청으로 확인되었는가
- 내부 자산 접근을 시도하지 않았는가

---

## 10. 인증/인가 우회 관점

목적:

- 서버가 클라이언트 입력값과 세션 값을 어떻게 신뢰하는지 확인한다.
- 권한 상승이나 실제 계정 탈취는 수행하지 않는다.

승인 후 검토 가능한 관점:

- Cookie 제거
- Authorization 제거
- 역할/권한 관련 파라미터 존재 여부
- userId, memberNo, accountId 등 객체 소유자 파라미터 존재 여부
- 일반 사용자와 관리자 요청 구조 차이
- 화면에 없는 기능의 직접 요청 가능성

금지:

- 실제 관리자 권한 탈취
- 타 사용자 세션 사용
- 운영 데이터 변경
- 대량 객체 접근
- 권한 없는 수정/삭제 실행

결과 정리 기준:

- 서버가 세션 기준으로 권한을 검증하는가
- 클라이언트가 보낸 role/userId 값을 신뢰하는가
- 실패해야 할 요청이 성공했는가
- 응답 메시지만 성공이고 실제 상태 변경은 없는지 구분했는가

---

## 11. Claude 사용 규칙

Claude는 이 문서를 사용할 때 다음 규칙을 따른다.

1. 이 문서는 기본 자동 테스트에 사용하지 않는다.
2. 우회 테스트가 필요하다고 판단되면 먼저 테스트 계획을 작성한다.
3. 테스트 계획에는 다음 항목을 포함한다.
   - 대상 요청
   - 관련 기능 흐름
   - 취약점 후보
   - 우회 테스트가 필요한 이유
   - 사용할 우회 관점
   - 예상 정상 응답
   - 취약할 경우 예상 응답
   - 위험도
   - 요청 횟수
   - 중단 조건
4. 사용자 승인 전에는 실제 요청을 전송하지 않는다.
5. 승인 후에도 승인된 요청, 승인된 파라미터, 승인된 우회 관점에 한해 단일 요청으로만 테스트한다.
6. 결과는 원본 응답과 변조 응답의 차이 중심으로 정리한다.
7. 우회 성공 여부를 단정하지 않고 근거 기반으로 판단한다.
8. 민감 데이터가 노출되면 즉시 중단한다.
9. 하나의 우회 관점이 확인되면 `variant_hunting.md` 기준으로 동일 패턴의 다른 기능을 가설화하되, 추가 테스트는 별도 승인 후 수행한다.
