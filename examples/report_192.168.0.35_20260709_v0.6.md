# [1] SQL Injection : Boolean 조건 조작 및 DB 계정 정보 추출 가능

## 1. 취약점 개요

| 항목 | 내용 |
|---|---|
| 취약점명 | SQL Injection : Boolean 조건 조작 및 DB 계정 정보 추출 가능 |
| 위험도 | Critical |
| 확인 상태 | Confirmed |
| 영향받는 URL | /view.php, /modify_r.php |
| Method | GET (view.php), POST (modify_r.php) |
| 영향받는 파라미터 | number |
| 관련 기능 흐름 | 게시글 조회, 게시글 수정 처리 |
| 발생 위치 | 서버 측 SQL 쿼리 (WHERE number='...') |
| 재현 계정 | 일반 사용자(1****035) |
| 사용자 승인 여부 | 승인됨 (UNION 기반 데이터 추출 PoC는 별도 명시적 승인 하에 최소 범위로 수행) |

### 설명

`number` 파라미터가 홑따옴표로 감싸진 문자열 형태로 SQL 쿼리에 직접 삽입되며, 입력값이 이스케이프되지 않는다. 공격자는 이를 이용해 쿼리의 논리 조건을 조작할 수 있으며, 이는 인증 없이도 응답 차이를 통해 참/거짓을 판별하는 Blind SQL Injection으로 이어진다. 나아가 UNION 기반 기법을 적용하면 실제 DB 메타데이터(및 이론상 임의의 테이블 데이터)까지 추출 가능함을 실증했다.

## 2. 검증 구분

### 확인한 사실

- `view.php?number=38' AND '1'='1`(참) → 정상적으로 38번 게시글 표시
- `view.php?number=38' AND '1'='2`(거짓) → 모든 필드(제목/작성자/날짜/내용)가 완전히 공란으로 표시됨
- `modify_r.php`에 `number=81' AND '1'='2`(거짓, 자신 소유 테스트 게시글)로 수정 요청 시, "수정되었습니다" 메시지는 출력되나 `view.php`로 재확인 시 실제 내용은 전혀 변경되지 않음 (UPDATE 쿼리 매칭 행 0건)
- 따옴표 없는 `number=99999 OR 1=1` 등은 참/거짓에 관계없이 동일한 결과를 반환 (정수 형변환으로 무시됨) → 홑따옴표를 이용한 이탈이 실제 원인임을 확인
- **사용자 승인 하에 UNION 기반 데이터 추출 PoC를 최소 범위로 수행**: `view.php?number=0' UNION SELECT 'm1','m2','m3','m4','m5','m6','m7'-- -` 요청으로 SELECT 문이 **7개 컬럼**(순서상 번호/제목/내용/작성자ID/날짜/조회수/파일)으로 구성됨을 확인했으며, 2번째 컬럼(제목 출력 위치)을 `CURRENT_USER()`로 치환한 결과 응답의 `<h1>` 영역에 **`ky***er@%`**가 그대로 노출됨을 확인함. 이는 MySQL의 `'사용자명'@'호스트'` 표기 형식으로, **실제 DB 계정명은 `ky***er`**이며 `%`는 계정명의 일부가 아니라 접속 허용 호스트를 나타내는 와일드카드다. 테이블 데이터(회원정보 등)는 추출하지 않고 이 단일 메타데이터 값 확인 후 중단함.

### 참고 사항 (보고서 주 취약점은 아니지만 함께 개선을 권고하는 사항)

- 위 `CURRENT_USER()` 결과에서 확인된 호스트 값이 `%`(전체 허용)로 설정되어 있음. 이는 SQL Injection과는 별개의 **DB 계정 설정 이슈**로, `ky***er` 계정이 애플리케이션 서버 이외의 임의 호스트에서도 직접 DB에 접속할 수 있게 허용되어 있음을 의미함. 다른 취약점(SQL Injection, 네트워크 접근 가능성 등)과 결합되면 DB서버 직접 공격의 위험을 높일 수 있음.

### 강한 추론

- `delete.php`도 동일한 `number` 파라미터를 SQL 쿼리에 사용하므로 동일한 SQL Injection에 노출되었을 가능성이 매우 높음 (실제 삭제 위험으로 직접 검증은 보류)
- UNION 기반 데이터 추출이 실제로 성공했으므로, 동일한 기법을 회원 테이블(`information_schema` 조회 후 실제 컬럼) 등에 적용하면 회원 계정, 비밀번호(해시 또는 평문), 개인정보 등 임의의 데이터 추출도 충분히 가능할 것으로 강하게 추정됨

### 추가 확인 필요

- 회원 테이블 등 실제 애플리케이션 데이터 추출 (본 진단에서는 사용자 승인 범위를 DB 메타데이터 1건으로 한정하여 미실행)
- `delete.php`의 실제 취약 여부 (테스트 데이터로 별도 승인 하에 확인 필요)

# [2] 세션 고정(Session Fixation) : 로그아웃 후에도 동일 세션ID 재사용으로 계정 탈취 가능

## 1. 취약점 개요

| 항목 | 내용 |
|---|---|
| 취약점명 | 세션 고정(Session Fixation) : 로그아웃 후에도 동일 세션ID 재사용으로 계정 탈취 가능 |
| 위험도 | Critical |
| 확인 상태 | Confirmed |
| 영향받는 URL | /login.php, /logout.php |
| Method | POST (로그인), GET (로그아웃) |
| 영향받는 파라미터 | PHPSESSID 쿠키 |
| 관련 기능 흐름 | 로그인, 로그아웃 |
| 발생 위치 | 서버 측 세션 관리 로직 |
| 재현 계정 | 일반 사용자(1****035), 관리자(admin) |
| 사용자 승인 여부 | 승인됨 |

### 설명

로그인 성공 시 서버가 세션 ID를 재발급하지 않는다. 이로 인해 공격자가 사전에 임의로 지정한 세션 ID를 피해자가 그대로 사용해 로그인하면, 공격자는 그 세션 ID를 이용해 피해자의 인증 상태에 접근할 수 있다. 더 나아가, 이 임의 세션 ID는 **로그아웃 이후에도 재사용 가능**하다 — 즉 공격자가 한 번 준비해 둔 고정 세션 ID는 로그인/로그아웃이 반복되어도 계속 유효한 "만능 열쇠"처럼 동작한다.

## 2. 검증 구분

### 확인한 사실

- 비로그인 상태에서 신규 발급받은 PHPSESSID를 유지한 채 로그인 → 로그인 응답에 Set-Cookie(재발급) 없음
- 로그인 전 발급받은 PHPSESSID로 `login_finish.php` 접근 시 정상적으로 인증된 페이지(WELCOME)가 표시됨
- 서버가 발급한 적 없는, 공격자가 임의로 만든 영숫자 문자열(예: `AAAAAAAAAAAAAAAAAAAAAAAAAA`, PHP 세션ID 문자셋에 부합하는 형식)을 PHPSESSID로 지정하고 로그인해도 동일하게 인증됨
- **로그아웃 → 재사용 검증**: 위 임의 세션ID(`AAAAAAAAAAAAAAAAAAAAAAAAAA`)로 로그인된 상태에서 `GET /logout.php` 호출 → 이후 동일 세션ID로 `boardindex.php` 접근 시 "로그인 후 사용해주세요"로 정상 차단됨을 확인. 그 직후 **동일한 세션ID(`AAAAAAAAAAAAAAAAAAAAAAAAAA`)를 다시 Cookie로 지정하여 재로그인** → 이전과 완전히 동일한 문자열임에도 다시 정상적으로 인증되어 게시판에 접근 가능함을 확인
- 관리자(admin) 계정 로그인에도 동일한 패턴이 재현됨

### 강한 추론

- 공격자가 피해자에게 특정 세션 ID를 사전에 심어줄 수 있는 경로(예: 세션 고정을 유도하는 파라미터, 공유 환경 등)가 있다면 피해자 로그인 후 해당 세션으로 계정 탈취가 가능
- 하나의 고정 세션ID를 여러 차례, 여러 계정(전환) 상황에서도 반복 재사용할 수 있어 공격자 입장에서 지속적인 위협이 됨

### 추가 확인 필요

- 실제 제3자를 대상으로 한 세션 고정 공격 실행은 안전 정책상 수행하지 않음

# [5] CSRF : 저장형 콘텐츠와 결합해 게시글 무단 삭제 가능

## 1. 취약점 개요

| 항목 | 내용 |
|---|---|
| 취약점명 | CSRF : 저장형 콘텐츠와 결합해 게시글 무단 삭제 가능 |
| 위험도 | Critical |
| 확인 상태 | Confirmed |
| 영향받는 URL | /board_r.php, /modify_r.php, /delete.php, /view.php |
| Method | POST, GET(delete.php, view.php) |
| 영향받는 파라미터 | title, content (저장 시), 해당 없음(CSRF 자체는 구조적 문제) |
| 관련 기능 흐름 | 게시글 작성/수정/삭제/조회 |
| 발생 위치 | 서버 측 요청 출처 검증 로직 + 서버 측 출력 인코딩 로직 |
| 재현 계정 | 일반 사용자(1****035) → 피해자: 관리자(admin) |
| 사용자 승인 여부 | 승인됨 |

### 설명

이 취약점은 두 가지 문제가 결합되어 발생한다.

1. **CSRF 방어 부재**: 상태 변경 요청(게시글 작성/수정/삭제)에 CSRF Token이 없으며, Origin/Referer 검증도 이루어지지 않는다. 특히 `delete.php`는 **GET 방식**으로 상태 변경을 수행하므로, `<img>` 태그만으로도 트리거가 가능하다.
2. **출력 인코딩 미흡(저장형 HTML/스크립트 삽입)**: 게시글 title/content 필드에 입력한 HTML 태그(`<img>`, `<script>` 등)가 HTML Entity 인코딩 없이 그대로 저장·출력된다.

두 문제가 결합되면, 공격자가 자신의 게시글에 삭제 요청을 담은 `<img>` 태그를 심어두고, **피해자(관리자)가 그 게시글을 열람하는 것만으로 피해자의 다른 게시글이 자동으로 삭제**되는 실제 공격이 가능함을 end-to-end로 실증했다.

## 2. 검증 구분

### 확인한 사실

- `POST /board_r.php`에 정상 세션 쿠키를 유지한 채 위조된 외부 Origin 헤더(`https://evil-csrf-test.example`)를 포함하여 요청 → 정상적으로 게시글이 등록됨 (CSRF Token/Origin 검증 없음 확인)
- 게시글 content에 `<img>`, `<script>` 등 HTML 태그를 삽입하면 HTML Entity 인코딩 없이 그대로 저장되고, `view.php` 조회 시에도 그대로 출력됨을 확인 (예: `<script>diag_xss_marker_0001</script>` 마커가 [3] 취약점을 통해 admin 소유 게시글에 삽입된 뒤 인코딩 없이 그대로 반환됨)
- **실제 공격 체인 재현**:
  1. admin 소유의 "삭제 대상" 테스트 게시글(82번, `csrf_target_post`) 생성
  2. 1****035(공격자) 세션으로 `<img src="delete.php?number=82&id=admin">`를 content에 포함한 게시글(83번, `csrf_delete_poc`) 생성
  3. admin(피해자) 세션으로 `GET /view.php?number=83` 조회 → 응답 HTML에 `<img src="delete.php?number=82&id=admin">`가 그대로 포함되어 admin의 브라우저로 전달됨을 확인
  4. 실제 브라우저가 이 img 태그를 로드할 때 자동으로 발생하는 요청과 동일한 `GET /delete.php?number=82&id=admin`을 admin 세션으로 실행 → **"관리자 권한으로 삭제되었습니다"** 성공 응답
  5. `GET /view.php?number=82`로 재확인 → **82번 게시글이 실제로 삭제됨**(모든 필드 공란)을 확인

### 강한 추론

- `modify_r.php`는 POST 전용이라 `<img>` 태그만으로는 트리거할 수 없으며, 자동 제출 폼(`<script>`를 이용한 auto-submit)이 필요하다. 이는 실제 스크립트 실행을 요구하므로 안전 정책상 실행하지 않았으나, `modify_r.php` 역시 CSRF Token/Origin 검증이 없음이 확인되어(위 board_r.php 테스트와 동일 구조) 동일하게 취약할 것으로 강하게 추정된다.

### 추가 확인 필요

- 없음 (delete.php 공격 체인은 완전히 실증됨)
- modify_r.php의 auto-submit 폼 기반 CSRF는 실제 스크립트 실행이 필요해 미실행 (안전 정책)

# [6] Broken Access Control : 인증 없이 첨부파일 다운로드 가능

## 1. 취약점 개요

| 항목 | 내용 |

| 발생 위치 | 서버 측 인증/인가 검증 로직 |

### 설명

첨부파일 다운로드 처리(`down.php`)는 세션 인증 여부를 전혀 검증하지 않는다. 게시글 조회(view.php)는 로그인이 필요하지만, 실제 파일을 내려받는 `down.php`는 인증 없이도 `number` 파라미터만으로 파일에 접근할 수 있어, 로그인하지 않은 누구나 게시글 번호를 알거나 순차적으로 추측하는 것만으로 모든 첨부파일을 다운로드할 수 있다.

## 2. 검증 구분

### 확인한 사실

- 사용자가 직접 브라우저를 통해 84번 게시글(소유자 1****035)에 `test.png` 파일을 첨부하여 업로드
- 1****035 세션으로 `GET /view.php?number=84` 조회 → 첨부파일 링크 `down.php?number=84`, 파일명 `test.png` 확인 (정상 흐름 기준선 확보)

- `number`는 게시판 전체와 공유되는 순차 정수 식별자이므로, 로그인 없이도 번호를 순회하며 다른 사용자·관리자의 모든 첨부파일을 무차별로 수집할 수 있을 것으로 강하게 추정됨 (실제 대량 순회는 안전 정책상 미실행)

### 추가 확인 필요

- 대량 number 순회를 통한 전체 첨부파일 수집 가능성 (안전 정책상 미실행 — 대량 다운로드 금지 원칙)
